## System 1: State File Writer

**Problem**: Multiple processes need to share state via files safely

**Solution**: Atomic file writes with optional locking

### Purpose
- Write JSON data atomically (all-or-nothing)
- Support file locking on platforms that have it
- Graceful degradation on platforms without locking
- Enable watchdog process to read bot state

### Atomic Write Strategy

#### Write to Temporary File First
1. Create temporary file with different name
2. Write data to temporary file
3. Flush to disk
4. Atomic rename to target filename

**Benefits**:
- Readers never see partial data
- Works on both major platforms
- If write fails, original file unchanged

### File Locking (Platform-Dependent) (need to create for both platforms because production environment is linux, and development environment is windows)

#### On Linux/Unix
- Acquire exclusive lock before writing
- Other processes wait for lock release
- Prevents concurrent write conflicts

#### On Windows  
- No locking available (library not present)
- Relies on atomic rename only
- Safe for single-process scenarios

### Operations Provided

#### Write JSON Atomic
- **Input**: Filename, data dictionary
- **Output**: File written to state directory
- **Format**: Formatted JSON (human-readable)

#### Append to JSON Array
- **Input**: Filename, new item, maximum size
- **Process**:
  1. Read existing array
  2. Append new item
  3. Trim to maximum size (oldest removed)
  4. Write back atomically
- **Use**: Append-only logs with size limit

#### Merge JSON Dictionary
- **Input**: Filename, updates dictionary
- **Process**:
  1. Read existing dictionary
  2. Merge in updates (overwrites keys)
  3. Write back atomically
- **Use**: Partial state updates

#### Flag File Operations
- **Write flag**: Create empty file
- **Remove flag**: Delete file if exists
- **Use**: Signal events (clean shutdown flag)

## System 2: Adaptive Scaling Manager

**Problem**: System load varies dramatically, need to adapt capacity

**Solution**: Multi-tier scaling with automatic tier selection

### Scaling Philosophy

#### Asymmetric Scaling
- **Scale-up**: Instant when threshold reached (no delays)
- **Scale-down**: Delayed with timer (prevent thrashing)
- **Hysteresis**: Different thresholds for up vs down

#### Why Asymmetric
- Load spikes need immediate response
- Load drops may be temporary (don't over-react)
- Prevents rapid up-down oscillation

### Tier System Structure

**Multi-tier configuration**: Each tier has:
- Queue capacity
- Batch size
- Processing interval
- Threshold (utilization trigger point)

**Growth pattern**: Each tier larger than previous
- Queue grows by multiplicative factor
- Batch size grows proportionally
- Interval stays constant across all tiers

**Tier levels**: Multiple tiers from baseline to maximum

### Scaling Decision Logic

#### Scale-Up Trigger
- **Condition**: Queue utilization exceeds upper threshold
- **Timing**: Immediate (no cooldown)
- **Action**: Move to next higher tier
- **Limit**: Can't exceed maximum tier

#### Scale-Down Trigger
- **Condition 1**: Timer expires (extended period without hitting threshold)
- **Condition 2**: Utilization below lower threshold
- **Timing**: Delayed (must wait for cooldown period)
- **Action**: Move to next lower tier
- **Limit**: Can't go below baseline tier

#### Error-Triggered Scaling
- **Condition**: Consecutive API failures exceed threshold
- **Action**: Scale down (reduce load on failing API)
- **Purpose**: Back off when external service struggling

### Hysteresis Design

**Upper threshold**: Higher percentage for scale-up
**Lower threshold**: Lower percentage for scale-down
**Gap**: Prevents oscillation at boundary

**Example**:
- Scale up at high utilization
- Don't scale down until much lower utilization
- Middle range: no scaling action

### State Tracking

#### Current State
- Current tier level
- Time entered current tier
- Last scale-up timestamp
- Last scale-down timestamp
- Last threshold hit timestamp

#### Historical Tracking
- Scaling event history (limited size)
- Time spent in each tier (cumulative)
- Total scale-up count
- Total scale-down count

### Scaling Event Structure

**Recorded for each scaling action**:
- Timestamp
- From tier and to tier
- Reason (threshold, timer, error)
- Queue size at scaling
- Utilization percentage

### Cooldown Management

**Scale-up cooldown**: None (instant response)
**Scale-down cooldown**: Moderate duration
**Purpose**: Prevent rapid consecutive scale-downs

### Configuration Callback

**Callback on scaling**: Notifies other systems of tier change
**Purpose**: Update queue size, batch size, interval
**Pattern**: Observer pattern for scaling events

### Anomaly Detection

**Checks for problematic patterns**:
- Stuck at high tier too long
- Excessive oscillation (too many scaling events)
- Rapid tier jumps (tier changes too quickly)
- High tier with many errors

**Purpose**: Alert to potential system issues

## System 3: Generic Memory Queue

**Problem**: Need thread-safe buffering with size limits

**Solution**: Generic queue with configurable overflow behavior

### Generic Design

**Type parameter**: Can store any data type
**Reusability**: Not specific to events
**Pattern**: Generic container with policies

### Thread Safety

**Lock mechanism**: Reentrant lock for thread safety
**All operations**: Protected by lock
**Concurrent access**: Safe from multiple threads

### Overflow Handling

#### Configurable Behavior
**Overflow handler**: Callback when overflow occurs
**Policy decision**: Handler determines what to do

#### Current Implementation
**On overflow**: Reject new item, preserve existing queue
**Rationale**: Lose latest data, not entire queue
**Alternative**: Could clear queue (more aggressive)

### Operations

#### Add Item
- **Acquire lock**
- **Check**: Queue at capacity?
- **If yes**: Call overflow handler, return false
- **If no**: Append item, increment counter, return true

#### Get Batch
- **Acquire lock**
- **Remove**: Up to requested count from front
- **Return**: List of items removed
- **Note**: May return fewer than requested

#### Size Query
- **Acquire lock**
- **Return**: Current item count

#### Clear
- **Acquire lock**
- **Remove**: All items
- **Return**: List of all items that were present

### Statistics Tracking

**Counters maintained**:
- Items added (lifetime)
- Items removed (lifetime)
- Overflow count
- Rejected item count

**Purpose**: Monitoring queue health

### Dynamic Capacity

**Update maximum size**:
- **Acquire lock**
- **Validate**: New size above minimum
- **Update**: Maximum size value
- **Effect**: Future adds use new limit

**Use case**: Adaptive scaling changes queue capacity

## System 4: Event Queue

**Problem**: Specialize generic queue for analytics events

**Solution**: Event-specific queue with utilization tracking

### Specialization

**Built on**: Generic memory queue
**Adds**: 
- Event validation
- Utilization calculation
- Threshold callbacks
- Event-specific statistics

### Threshold Callback System

**Purpose**: Trigger scaling when queue filling

**Mechanism**:
- Calculate utilization on each add
- If exceeds threshold: call callback
- Callback receives queue size and utilization
- No cooldown: immediate every time

**Use**: Enables instant scale-up response

### Overflow Callback

**Different from threshold**: Fires on overflow, not utilization
**Purpose**: Alert to data loss
**Payload**: Number of events rejected, queue statistics

### Event Validation

**Before adding**:
- Check event has required fields
- Reject invalid events
- Return false for invalid

**Required fields**: Event name, properties dictionary

### Utilization Tracking

**Calculation**: Current size ÷ maximum size
**Range**: From zero (empty) to one (full)
**Use**: Scaling decisions, monitoring

### Statistics

**Beyond generic queue**:
- Current utilization
- Total overflow events (cumulative)
- Max size (current limit)

### Dynamic Sizing Integration

**Propagates to generic queue**:
- Update max size when tier changes
- Logs size change
- Thread-safe operation

## System 5: Batch Coordinator

**Problem**: Collect events into batches and transmit to API

**Solution**: Timing-based and size-based batch coordinator

### Coordination Responsibilities

1. **Decide when**: Time-based and size-based triggers
2. **Collect batch**: Pull from queue
3. **Size checking**: Verify batch within limits
4. **Transmit**: Send to API via HTTP client
5. **Handle response**: Success/failure processing
6. **Update config**: Accept new batch parameters from scaling

### Batch Timing Logic

**Two triggers**:
1. **Timeout reached**: Interval since last batch expired
2. **Size threshold**: Queue reached batch size

**Also consider**:
- Memory threshold: Queue approaching capacity
- Immediate trigger: Queue critically full

**First trigger wins**: Whichever condition met first

### Batch Collection

**Process**:
1. Check queue has events
2. Request batch of configured size from queue
3. Verify batch within size limits
4. If too large: split batch, return portion to queue
5. Return batch for transmission

### Size Limit Protection

**Estimate batch size**: Before transmission
**Check against limit**: API size constraint
**If over limit**: Split batch in half

**Purpose**: Prevent API rejection

### Batch Transmission

**Async operation**:
1. Capture start time
2. Estimate batch size
3. Call HTTP client send method
4. Capture end time
5. Calculate response time
6. Process API response

**Result structure**:
- Success flag
- Events sent/failed count
- Response time
- Current tier
- Batch size in bytes
- Error message (if failed)

### Response Processing

**Success path**:
- Increment success counters
- Update statistics
- Call success callback

**Failure path**:
- Increment failure counters
- Update statistics  
- Call failure callback

**Callback pattern**: Notify monitor of result

### Configuration Updates

**Dynamic reconfiguration**:
- New batch size
- New interval
- Current tier

**Thread-safe**: Lock-protected updates
**Logged**: Configuration changes printed

### Statistics

**Tracked metrics**:
- Batches sent (total)
- Successful batches
- Failed batches
- Total events sent
- Success/failure rates
- Average batch sizes

## System 7: Processing Coordinator

**Problem**: Coordinate event processing pipeline

**Solution**: Bridge between processor and queue

### Responsibilities

1. Receive raw event data
2. Call event processor
3. Handle processor result (may be multiple events)
4. Queue all resulting events
5. Track processing statistics

### Processing Flow

**Input**: Event type and raw event data
**Process**:
1. Call event processor's main method
2. Receive list of processed events
3. For each processed event:
   - Add to event queue
   - Track success/failure
4. Update statistics

**Output**: Boolean indicating overall success

### Statistics

**Counters**:
- Total processed
- Successful
- Failed

**Thread-safe**: Lock-protected

### Error Handling

**Processor errors**: Caught and logged
**Queue errors**: Caught and logged
**Statistics updated**: Even on failure

## Key Design Decisions

### Why Asymmetric Scaling
- Load spikes need immediate response (scale up fast)
- Load drops may be temporary (scale down slow)
- Prevents thrashing between tiers

### Why Constant Interval
- Simplifies configuration
- Predictable batch timing
- Only queue and batch size change

### Why Preserve Queue on Overflow
- Original: cleared entire queue (catastrophic loss)
- Fixed: reject only new event (minimal loss)
- Better data preservation

### Why Category-Based Tracking
- Automatic discovery of new channels
- No hardcoded channel lists
- Scales with server growth

### Why State Files
- Enables independent watchdog process
- Survives process crashes
- Human-readable for debugging
- No database dependency

### Why Multiple Statistics Levels
- Generic queue: Basic buffering stats
- Event queue: Utilization and validation
- Batch coordinator: Transmission stats
- Monitor: Comprehensive system stats
- Each level appropriate granularity
