# Event Processing Logic Module - Technical Concepts Guide

**Build Order: #3 (Business Logic Core)**  
**Dependencies**: Event schema configuration file (reads YAML)  
**Purpose**: Transform platform events into localized analytics events with business rules

---

## Module Purpose

This module implements **business logic** that:
- Receives raw platform events (user actions)
- Applies domain-specific identification rules
- Generates localized event names
- Correlates bot interactions with user triggers
- Validates against schema definitions
- Produces structured analytics events

**Core Responsibility**: "What happened?" → "What does this mean in our business context?"

## Business Rule Layer: Category Trackers

### Forum Category Tracker

**Problem**: Forums are added/removed dynamically, don't want to hardcode every forum

**Solution**: Track by parent category membership

#### Tracked Categories Storage
- Set of category identifiers
- Automatically includes all forums in those categories

#### Category Name Cache
- Maps category identifier to category name
- Performance optimization (avoid repeated lookups)

#### Membership Check
- **Input**: Category identifier
- **Returns**: Boolean indicating if tracked
- **Logic**: Check if identifier in tracked set

#### Cache Operations
- Store category name for identifier
- Retrieve category name for identifier
- **Purpose**: Display friendly names in events

### Voice Category Tracker

**Problem**: Distinguish fixed server channels from user-created temporary channels

**Solution**: Track specific category for fixed channels

#### Tracked Categories Storage
- Set of category identifiers for fixed channels
- Anything NOT in this set is custom

#### Binary Classification
- **In tracked category**: Fixed voice channel
- **Not in tracked category**: Custom voice channel
- **Purpose**: Different event names and flow categories

#### Same Cache Pattern
- Store and retrieve category names
- Performance optimization

**Key Insight**: Absence from set is meaningful (custom detection)

## Business Rule Layer: Specialized Handlers

### Role Change Handler

**Problem**: Role changes need special processing (may trigger multiple events)

**Solution**: Unified handler that generates appropriate events

#### Constants
- Flow category for role events
- Maximum role name length (truncation threshold)
- Event naming patterns for additions/removals
- Special marketing roles set
- Special event configuration for marketing roles

#### Main Processing Method
- **Input**: Event type, role name, additional context
- **Output**: LIST of event identifications (can be multiple)
- **Logic**:
  1. Generate standard role change event
  2. Check if marketing role
  3. If marketing role: generate SECOND special onboarding event
  4. Return list of events

#### Marketing Role Detection
- **Check**: Role name in marketing roles set
- **Result**: Dual event generation
- **Purpose**: Track onboarding milestone

#### Event Name Formatting
- **Input**: Role name, action type
- **Output**: Localized event name
- **Handles**: Truncation if role name too long
- **Pattern**: Different templates for add vs remove

### General Chat Handler

**Problem**: Main chat channel needs different event names than other channels

**Solution**: Channel-specific detection and naming

#### Configuration
- Specific channel identifier for general chat
- Event name mappings (message type → event name)

#### Detection Method
- **Input**: Channel identifier
- **Check**: Exact match with configured identifier
- **Returns**: Event identification or null

#### Event Mapping
- Message type: specific event name
- GIF type: specific event name
- Emoji type: specific event name
- Sticker type: specific event name

### Mod Message Handler

**Problem**: Track when moderators update important channels

**Solution**: Channel and sender whitelist

#### Configuration
- Map of channel identifiers to configuration
- Per-channel: authorized sender identifiers, event name

#### Detection Method
- **Input**: Channel identifier, sender identifier
- **Check 1**: Channel in tracked map
- **Check 2**: Sender in authorized list for that channel
- **Returns**: Event identification or null

#### Event Generation
- **Uses**: Pre-configured event name from map
- **Category**: Onboarding flow
- **Purpose**: Track important content updates

### Special Thread Handler

**Problem**: Some threads need unique tracking regardless of bot correlation

**Solution**: Thread identifier mapping

#### Configuration
- Map of thread identifiers to event configuration
- Per-thread: event name, flow category

#### Detection Method
- **Input**: Thread identifier, interaction type
- **Check**: Thread in special thread map
- **Returns**: Event identification or null

#### Use Case
- Threads for specific activities
- Not tied to bot correlation
- Special naming and categorization

### Room Creation Handler

**Problem**: Detect when user triggers room creation feature

**Solution**: Specific voice channel detection

#### Configuration
- Single channel identifier for creation trigger

#### Detection Method
- **Input**: User identifier, from channel, to channel
- **Check**: Either joining OR leaving trigger channel
- **Returns**: Event identification or null

#### Dual Detection
- Joining trigger channel: room creation
- Leaving trigger channel: also room creation
- **Reason**: Different platform behaviors

### Generic Channel Processor

**Problem**: Handle patterns for voice/forum/text channels uniformly

**Solution**: Configurable pattern processor

#### Configuration Per Channel Type
- Event patterns (interaction type → name template)
- Flow category (static or pattern)
- Flow category pattern (template if dynamic)
- Category tracker reference (for dynamic discovery)

#### Pattern Detection Method
- **Input**: Channel identifier, name, interaction type, category identifier, channel type
- **Process**:
  1. Look up event pattern for interaction type
  2. Format event name with channel context
  3. Determine flow category (static or dynamic)
  4. Check category membership if applicable
  5. Build additional properties
- **Returns**: Event identification or null

#### Category-Based Flow Selection
- Voice channels: Check if in fixed category → different flow
- Forums: Use forum name in flow category
- Text: Use static flow category

## Correlation Layer: Bot Interaction Tracking

### Bot Configuration Structure

**Problem**: Different bots need different tracking strategies

**Solution**: Per-bot configuration map

#### Tracked Bots Registry
- Map of bot identifiers to configuration
- Per-bot configuration includes:
  - Bot name and localized name
  - Event name pattern
  - Flow category
  - Thread identifier (if thread-based)
  - Correlation flag (use correlation or not)

#### Correlation Settings
- Correlation time window
- Minimum confidence threshold
- Pending message timeout duration
- Specific thread identifier for special tracking

### Message Buffer System

**Problem**: Need to find recent user messages when bot responds

**Solution**: Per-channel message buffer

#### Channel Buffer Structure
- Map of channel identifiers to buffer
- Each buffer stores recent user messages
- Limited size (oldest removed when full)

#### Buffer Operations
- **Add message**: Store user message with timestamp
- **Find triggering user**: Search for message within time window
- **Lookup logic**: Reverse chronological search (most recent first)

### Correlation Engine

**Problem**: Link bot responses to user actions

**Solution**: Time-based correlation with confidence scoring

#### Correlation Method
- **Input**: Bot identifier, channel identifier, bot response time
- **Process**:
  1. Check if bot uses correlation
  2. Retrieve channel's message buffer
  3. Search for user message within time window
  4. Calculate confidence score
  5. Return correlation result
- **Output**: Correlation result structure

#### Confidence Calculation
- **Based on**: Time difference between user message and bot response
- **Formula**: Closer in time = higher confidence
- **Range**: From zero confidence to full confidence
- **Threshold**: Below minimum, don't correlate

#### Correlation Decision
- **Correlated**: Found user message with sufficient confidence
- **Uncorrelated**: No message found or confidence too low
- **Type**: Descriptive string indicating correlation status

### Pending Message Timeout System

**Problem**: User messages bot but bot doesn't respond

**Solution**: Track pending messages with timeout

#### Pending Message Registry
- Map of message identifiers to pending data
- Stores: user info, thread info, message content, timestamp

#### Timeout Task Management
- Map of message identifiers to async tasks
- Tasks run timeout check after delay
- Cancellation when bot responds

#### Add Message With Tracking
- **Process**:
  1. Add to correlation buffer (normal)
  2. Check if message in special tracked thread
  3. If yes: mark as pending with timeout data
  4. Schedule timeout task
- **Purpose**: Detect no-response scenario

#### Cancel Pending Timeout
- **Trigger**: Bot responds to user
- **Process**:
  1. Find user's pending message
  2. Cancel timeout task
  3. Remove from pending registry
- **Purpose**: Prevent false uncorrelated event

#### Generate Uncorrelated Event
- **Trigger**: Timeout expires without bot response
- **Process**:
  1. Retrieve pending message data
  2. Create event identification with thread context
  3. Remove from pending registry
  4. Cancel timeout task
- **Output**: Event identification for uncorrelated message

#### Thread Name Importance
- **Used**: Generate event name "Message in {thread_name}"
- **Source**: Thread name from pending data
- **Purpose**: Context-specific uncorrelated events

### Bot Response Processing

**Problem**: Integrate correlation with event naming

**Solution**: Bot-specific event generation with correlation data

#### Event Naming Strategy
- **Base**: Bot's configured event name
- **Suffix**: Added based on thread location (if correlation bot)
- **Logic**: Simple thread identifier comparison
- **Result**: Event name with location context

#### Suffix Determination
- **Check**: Bot response thread identifier
- **Compare**: Against specific tracked thread
- **Result**: Binary suffix (in thread or out of thread)
- **Purpose**: Different event names for different contexts

#### Correlation Enrichment
- **If correlated**: Add triggering user information to properties
- **If not correlated**: Add correlation failure information
- **Both**: Add correlation metadata (type, confidence, delay)

## Orchestration Layer: Event Identifier

**Problem**: Coordinate all handlers to identify any event type

**Solution**: Main coordinator that tries handlers in priority order

### Identification Strategy

Three-phase approach:

#### Phase 1: Specific Event Identification
Try specialized handlers in order:
- Room creation detection (highest priority)
- Voice channel switching
- Thread following (forum threads only)
- Announcement following
- Thread creation/deletion (forum threads only)
- Bot thread message detection
- Mod message detection
- Role change events
- General chat events
- Special thread events

**Early exit**: First match wins, return immediately

#### Phase 2: Pattern-Based Identification
If no specific match:
- Determine channel category (voice/forum/text)
- Map event type to interaction type
- Call generic channel processor
- Use category-based detection

**Fallback**: Generic pattern matching

#### Phase 3: Fallback Event
If still no match:
- Create generic unidentified event
- Mark as unidentified
- Include original event type
- Preserves data even if can't classify

### Bot Message Processing

**Special path**: Bot messages handled separately

#### Detection
- Check if author is bot
- Get bot identifier from message

#### Process
- Retrieve bot configuration
- Generate base event name
- Apply suffix logic if correlation bot
- Correlate with user message
- Enrich with correlation data
- Return event identification + correlation result

### User Message Tracking

**Side effect**: All user messages tracked for correlation

#### Process
- Extract user and message details
- Add to correlation buffer
- Check if in special tracked thread
- If yes: mark as pending, schedule timeout
- Store timeout task reference

## Orchestration Layer: Event Processor

**Problem**: Main entry point that coordinates everything

**Solution**: High-level processor that manages full pipeline

### Initialization
- Load event schema configuration (YAML file)
- Initialize category trackers
- Create channel processors (voice, forum, text)
- Initialize event identifier
- Create handler instances
- Set up pending timeout task tracking

### Main Processing Method

**Signature**: Process platform event, return analytics events

**Input**:
- Event type
- User object (optional)
- Channel object (optional)
- Message object (optional)
- Additional data

**Output**: LIST of analytics events (can be multiple)

#### Processing Steps

1. **Data Extraction**
   - Extract user properties (if user present)
   - Extract channel properties (if channel present)
   - Extract message properties (if message present)
   - Combine with additional data

2. **User Message Tracking**
   - If user message: add to correlation buffer
   - If in special thread: mark as pending
   - Schedule timeout task

3. **Bot Detection**
   - Check if user is bot
   - If bot: take bot processing path
   - Correlate, generate events, return

4. **Event Identification**
   - Call event identifier with extracted data
   - Get list of event identifications
   - (Can be multiple for role events)

5. **Event Creation**
   - For each identification:
     - Create base event structure
     - Validate against schema (optional)
     - Create analytics event format
   - Return list of analytics events

### User Data Extraction

**Purpose**: Normalize platform user object to standard properties

#### Extracted Fields
- User identifier (converted to string)
- Display name
- Is bot flag
- Server-specific display name (if available)
- Server identifier (if available)
- Role names (if available)

#### Identifier Safety
- All platform identifiers converted to strings
- Prevents precision loss in JSON serialization

### Channel Data Extraction

**Purpose**: Normalize platform channel object to standard properties

#### Extracted Fields
- Channel identifier (string)
- Channel name
- Channel type
- Server identifier
- Category identifier (if exists)
- Category name (if exists)

#### Thread Detection
- Check if channel has parent (is thread)
- Extract parent channel information
- Extract parent category information
- Include parent context in properties

#### Category Information
- Extract category identifier from channel
- Extract category object
- Get category name
- Store in properties for handler use

### Message Data Extraction

**Purpose**: Normalize platform message object to standard properties

#### Extracted Fields
- Message identifier (string)
- Message type
- Content character count
- Attachment presence and count
- Embed presence and count
- Sticker presence and count
- GIF detection

#### Thread Context
- Extract thread identifier if message in thread
- Extract thread name (critical for uncorrelated events)
- Include in message properties

#### Enhanced Content Detection
- Sticker information (identifiers, names, custom flag)
- Attachment information (identifiers, file upload flag)
- GIF detection from content

### Event Creation Pipeline

#### Base Event Creation
- **Input**: Event identification, context data
- **Output**: Standardized event structure
- **Combines**: Identification properties + context data
- **Adds**: Timestamp, trackability flags

#### Analytics Event Creation
- **Input**: Localized event name, properties
- **Output**: Platform-specific event format
- **Adds**: 
  - Unique identifier
  - User identifier (distinct_id)
  - Platform-specific time field
  - Insert identifier for deduplication

#### Distinct ID Generation
- **Preference**: Server display name
- **Fallback**: User identifier
- **Last resort**: System-generated identifier
- **Purpose**: User tracking across events

### Schema Validation

**Optional step**: Validate against loaded schema

#### Validation Process
- Check required properties present
- Verify property types
- Check property values in allowed ranges
- Return validation errors (empty if valid)

#### Error Handling
- Log validation failures
- Can choose to reject or allow invalid events
- Helps catch data quality issues

### Batch Processing Support

**Additional method**: Process multiple events efficiently

#### Process
- Loop through event list
- Call main processing method for each
- Collect all results
- Handle errors gracefully (skip bad events)
- Return combined results

### Timeout Task Processing

**Side effect management**: Handle completed timeout tasks

#### Cleanup Method
- Check pending timeout tasks
- Identify completed tasks
- Generate uncorrelated events from completed
- Remove completed from tracking
- Return generated events

#### Integration
- Called periodically by main application
- Ensures timeout events eventually fire
- Prevents memory leak from accumulated tasks

## Design Patterns Used

**Strategy Pattern**:
- Different handlers for different event types
- Event identifier coordinates strategies

**Chain of Responsibility**:
- Try specific handlers in sequence
- First match handles the event

**Factory Pattern**:
- Event identification creates event structures
- Different structures for different events

**Template Method**:
- Common extraction pattern for user/channel/message
- Specific implementations for each platform object

**Builder Pattern**:
- Event creation builds complex structures incrementally
- Combines identification + extraction + formatting

**Observer Pattern** (implicit):
- Timeout tasks observe message pending state
- Fire when timeout occurs

**Registry Pattern**:
- Bot configuration registry
- Pending message registry
- Timeout task registry

## Key Technical Decisions

### Why Multiple Events Per Input
- Role changes can trigger both role event and onboarding event
- Dual tracking for certain scenarios
- Richer analytics without data duplication

### Why String Identifiers
- Platform identifiers are very large integers
- JSON parsers may lose precision
- String prevents data corruption

### Why Correlation Window
- Too long: false correlations
- Too short: miss valid correlations
- Balance accuracy vs coverage

### Why Pending Timeouts
- Detect when users try to interact but bot doesn't respond
- Different meaning than successful interaction
- Captures incomplete interactions

### Why Category-Based Tracking
- Automatic discovery of new channels
- No code changes when channels added
- Scales better than hardcoded lists

### Why Schema Validation Optional
- Performance vs data quality trade-off
- Can validate in development, skip in production
- Fail-fast vs fail-safe choice
