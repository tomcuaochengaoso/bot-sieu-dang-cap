## Main Application Class

**Purpose**: Single class that owns entire application lifecycle

### Initialization Phase

**Configuration Loading**:
- Read environment variables for credentials
- Validate all required configuration present
- Extract API endpoints and tokens
- Set up paths for state files

**Component Creation Order** (Critical):
1. State file writer (file I/O utilities)
2. Event processor (business logic)
3. Transmission statistics tracker
4. HTTP client (API communication)
5. Scaling manager (adaptive scaling)
6. Event queue (with callbacks)
7. Batch coordinator (with dependencies)
8. Infrastructure monitor (monitoring facade)
9. Processing coordinator (event pipeline)
10. Platform client (event listener)

**Why This Order**:
- No dependencies → dependencies → dependents
- Bottom-up construction
- Each component has what it needs before initialization

### Dependency Wiring

**Problem**: Components need to communicate

**Solution**: Callback pattern for loose coupling

#### Callbacks to Wire:

**Queue Overflow → Monitor**:
- Queue calls monitor when overflow occurs
- Monitor writes alert trigger
- Decoupled: Queue doesn't know about alerts

**Batch Transmission → Monitor**:
- Coordinator calls monitor on transmission result
- Monitor records success/failure
- Decoupled: Coordinator doesn't know about monitoring

**Threshold Reached → Scaling Check**:
- Queue calls when utilization threshold hit
- Triggers immediate scaling evaluation
- Enables fast scale-up response

**Scaling Event → Configuration Update**:
- Scaling manager calls when tier changes
- Application updates queue size and batch config
- Propagates new settings to all components

**Wiring Method**:
- Set callback properties on components
- Use lambda or method references
- Indirect coupling through callbacks

## Callback Implementation

### Scaling Event Callback

**Trigger**: When scaling manager changes tier

**Process**:
1. Write scaling state via monitor
2. Get new tier configuration
3. Update queue maximum size
4. Update batch coordinator configuration (size, interval, tier)

**Purpose**: Propagate scaling changes to all affected components

### Threshold Callback

**Trigger**: Queue utilization exceeds threshold

**Process**:
1. Get current timestamp
2. Call scaling manager's check method
3. If scaling needed, scaling callback handles it
4. Provides immediate scale-up (no waiting for batch loop)

**Purpose**: Fast response to load spikes

### Transmission Callback

**Trigger**: Batch transmission completes (success or failure)

**Process**:
1. Record result in transmission statistics
2. Record in monitor (success or failure path)
3. Monitor may write alert if consecutive failures

**Purpose**: Track API health and trigger alerts

## Batch Processing Loop

**Problem**: Need to continuously collect and send events

**Solution**: Async loop with multiple responsibilities

### Loop Structure

**Infinite loop** with periodic sleep:
- Check if should send batch
- Send batch if ready
- Process completed timeout tasks
- Update monitoring data
- Check scaling conditions
- Display status periodically
- Sleep for current interval

### Responsibilities

#### 1. Heartbeat Writing
- Every loop iteration
- Proves process is alive
- Updates all current metrics

#### 2. Tier Time Tracking
- Calculate elapsed since last update
- Update scaling manager's tier time
- Reset tracking timestamp
- Purpose: Accurate tier distribution statistics

#### 3. Daily Stats Writing
- Periodic (every N iterations)
- Write comprehensive statistics
- Check for scaling anomalies
- Purpose: Data for daily summary email

#### 4. Scaling Check
- Check queue size
- Call scaling manager's check method
- Handles scale-down based on timer
- Note: Scale-up handled by threshold callback

#### 5. Batch Collection and Sending
- Check if should send (time or size based)
- Collect batch from queue
- Analyze event content
- Update monitor with event metrics
- Send batch via coordinator
- Transmission callback handles result

#### 6. Timeout Task Processing
- Check for completed pending message timeouts
- Generate uncorrelated events from completed tasks
- Add to queue
- Clean up completed tasks
- Purpose: Handle bot non-response scenarios

#### 7. Status Display
- Periodic (every N iterations)
- Show current metrics
- Display queue, tier, batch config
- Purpose: Operator visibility

### Event Analysis

**Before sending batch**:
- Generate summary statistics
- Count events by type
- Extract flow categories
- Track correlation rates
- Purpose: Monitoring and debugging

### Timeout Task Processing Details

**Check completed tasks**:
- Iterate through pending timeout tasks
- Identify tasks that finished
- Extract event identification results
- Create complete events
- Add to queue
- Record in monitor
- Clean up task references

**Purpose**: Ensure timeout-triggered events eventually fire

### Status Display Format

**Displayed information**:
- Current tier level
- Queue size and utilization
- Batch size and interval configuration
- Total events processed
- Pending timeouts count
- Uptime duration
- Consecutive API failures

**Purpose**: Quick health check without logs

## Application Lifecycle Management

### Startup Sequence

**Order of operations**:
1. Print startup banner with feature announcements
2. Set up signal handlers (graceful shutdown)
3. Start batch processing loop (async task)
4. Connect to platform (blocking, runs until shutdown)

**Signal Handling**:
- Register handlers for termination signals
- On signal: Set running flag to false
- Cancel batch processing task
- Close platform connection
- Note: Windows doesn't support signal handlers (use Ctrl+C)

### Running State

**Main event loop**:
- Platform client runs (blocking)
- Batch processing loop runs (async task)
- Both coordinate through queue
- Monitor writes state periodically
- Watchdog reads state independently

### Shutdown Sequence

**Graceful shutdown steps**:
1. Set running flag to false
2. Write clean shutdown flag (prevents crash alert)
3. Cancel all pending timeout tasks
4. Cancel batch processing task
5. Process any remaining completed timeouts
6. Write shutdown summary via monitor
7. Flush remaining events in queue
8. Send final batch if events remain
9. Close platform connection
10. Display final statistics

**Critical**: Clean shutdown flag tells watchdog "not a crash"

## Error Handling Strategy

### Component Initialization Errors

**Fatal errors** (abort startup):
- Missing credentials
- Configuration errors
- Module import failures

**Log and exit**: Can't run without core components

### Runtime Errors

**Batch processing errors**:
- Catch in loop
- Log error
- Continue running
- Sleep briefly (prevent tight error loop)

**Event processing errors**:
- Caught in processor
- Logged
- Event skipped
- Continue processing

**Transmission errors**:
- Caught in coordinator
- Recorded as failure
- May trigger scaling
- May trigger alert
- Continue running

### Shutdown Errors

**Best effort**:
- Try to write final state
- Try to send final batch
- Don't crash during shutdown
- Purpose: Clean up as much as possible

## Async/Await Coordination

### Main Event Loop

**Platform client**: Blocking call
- Runs in main async function
- Returns when disconnected

**Batch processor**: Async task
- Created and scheduled
- Runs concurrently with platform
- Cancelled on shutdown

### Task Management

**Batch task**:
- Created with `asyncio.create_task`
- Stored for cancellation
- Awaited during shutdown (with cancellation)

**Timeout tasks**:
- Created per pending message
- Stored in dictionary
- Cancelled when bot responds
- Cleaned up when completed

**Sleep operations**:
- Use `asyncio.sleep` (non-blocking)
- Allows other coroutines to run
- Cooperative multitasking

## Platform-Specific Considerations

### Signal Handling

**Unix-like systems** (Linux, Mac):
- Register signal handlers
- SIGTERM and SIGINT handled
- Clean shutdown on signals

**Windows**:
- Signal handlers not supported
- Keyboard interrupt (Ctrl+C) works
- Graceful degradation

### File Operations

**State files**:
- Cross-platform paths (pathlib)
- Atomic writes
- File locking on Linux (optional on Windows)

## Key Design Decisions

### Why Single Application Class

**Simplicity**:
- One place to understand lifecycle
- Clear ownership of components
- Easy to test (mock components)

**Alternative**: Could split into multiple managers
**Trade-off**: More classes vs. larger single class

### Why Callback Wiring

**Loose Coupling**:
- Components don't import each other
- Easier to test in isolation
- Easier to replace components

**Alternative**: Direct method calls
**Trade-off**: Flexibility vs. simplicity

### Why Batch Loop Not Platform Event

**Separation**:
- Platform events: external events
- Batch processing: internal orchestration
- Don't mix concerns

**Independence**:
- Batch timing independent of event rate
- Can run even when no events

### Why Monitor Facade

**Encapsulation**:
- Hide state file complexity
- Single interface for monitoring
- Easy to change implementation

**Consistency**:
- All monitoring through same interface
- Easier to add new monitoring

### Why Clean Shutdown Flag

**Problem**: Watchdog can't distinguish crash from clean shutdown

**Solution**: Write flag file on shutdown
- Watchdog checks for flag
- Flag present: Skip crash alert
- Flag absent: Send alert

**Alternative**: Could use exit code (not visible to watchdog)

## Build Your Own: Key Questions

When building your orchestrator, ask:

1. **What order must components initialize?** (dependency graph)
2. **How will components communicate?** (callbacks, events, queues)
3. **What happens on startup failure?** (fail fast or degrade gracefully)
4. **What happens during shutdown?** (data loss prevention)
5. **How to prove system is alive?** (heartbeat, health checks)
6. **How to handle varying load?** (scaling strategy)
7. **How to monitor health?** (metrics, alerts, summaries)

Answer these questions to design your orchestration layer.
