# Alert Manager (Watchdog) Module - Technical Concepts Guide

**Build Order: #6 (Independent Monitoring)**  
**Dependencies**: Gmail API library (for email alerts), State file reader (reads bot state)  
**Purpose**: Independent health monitoring and email alerting

---

## Module Purpose

This is an **independent watchdog process**:
- Runs separately from main bot (different process)
- Monitors bot health via state files
- Detects crashes (stale heartbeat)
- Sends email alerts via Gmail API
- Checks for anomalies (queue overflow, scaling issues)
- Generates daily summary emails

**Core Responsibility**: "Is the bot healthy? If not, alert the human."

---

## Why Independent Process

### Problem
If monitoring runs in same process as bot, when bot crashes, monitoring dies too

### Solution
Separate process that:
- Survives bot crashes
- Reads bot state from disk (state files)
- Can detect when bot stops updating heartbeat
- Alerts human via email (not platform messaging)

**Key insight**: Watchdog must outlive the thing it watches

---

## Architecture Overview

The module contains several independent systems:

1. **Gmail Email Sender** - OAuth2 email via Gmail API
2. **Alert Rate Limiter** - Prevent email spam
3. **State File Reader** - Read bot state files
4. **Bot Health Monitor** - Detect crashes via heartbeat
5. **Scaling Anomaly Detector** - Find scaling problems
6. **Daily Summary Collector** - Gather statistics
7. **Alert Manager** - Main orchestrator

## System 1: Gmail Email Sender

**Problem**: Need reliable email delivery for alerts

**Solution**: Use Gmail API with OAuth2 authentication

### Authentication Strategy

**OAuth2 flow**:
1. Load credentials from JSON file
2. Load existing token if available
3. If token expired: refresh using refresh token
4. If no token: Run OAuth flow in browser
5. Save token for next run

**Benefit**: No password in code, secure authentication

### Token Management

**Token file**: Stores authentication token
**Refresh**: Automatic when expired
**First run**: Interactive browser authentication
**Subsequent runs**: Use saved token

### Email Sending

**HTML email support**: Rich formatting
**Priority setting**: Normal, high, critical
**Retry logic**: Multiple attempts on failure
**Success confirmation**: Log send confirmation

### Email Formatting

**HTML template system**:
- Header with title and branding
- Multiple sections with styling
- Color-coded sections (critical, warning, success, info)
- Statistics formatting (label-value pairs)
- Tier badges (color-coded by tier level)
- Tables for structured data
- Footer with timestamp

**Styling**:
- Professional appearance
- Color coding for urgency
- Responsive layout
- Monospace for technical data

### Retry Logic

**Attempts**: Multiple retries with backoff
**Exponential delay**: Wait longer between retries
**HTTP error handling**: Specific handling for API errors

## System 2: Alert Rate Limiter

**Problem**: Don't spam inbox with duplicate alerts

**Solution**: Cooldown tracking per alert type

### Rate Limiting Strategy

**Per-alert-type cooldowns**:
- Queue overflow: Moderate cooldown
- API failure: Longer cooldown
- Bot crash: Short cooldown
- Scaling anomaly: Long cooldown

**Rationale**: Different alerts have different urgency

### State Persistence

**State file**: Stores last alert timestamps
**Survives restarts**: Read state on startup
**Atomic updates**: Save after each alert

### Cooldown Check

**Process**:
1. Check last alert time for type
2. Calculate elapsed time
3. If elapsed < cooldown: deny
4. If elapsed >= cooldown: allow

**Returns**: Boolean indicating if can send

### Alert Recording

**Process**:
1. Record current timestamp
2. Increment alert count for type
3. Save state to file

**Purpose**: Track alerting patterns

## System 3: State File Reader

**Problem**: Safely read state files written by bot

**Solution**: Robust file reading with error handling

### File Reading Strategy

**JSON parsing**:
- Try to read and parse JSON
- Handle missing files gracefully
- Handle corrupted JSON gracefully
- Return None on any error

**No locking required**: Read-only, atomic writes from bot

### State Files Read

1. **Bot heartbeat**: Current bot status
2. **Daily stats**: Statistics for summary
3. **Alert triggers**: Pending alerts to process
4. **Scaling state**: Tier history and distribution

### Heartbeat File Reading

**Contains**:
- Last update timestamp
- Events processed count
- Queue size and max
- Uptime duration
- API failure count
- Current tier
- Batch configuration

**Use**: Detect if bot alive and healthy

### Daily Stats File Reading

**Contains**:
- Total events processed
- Luong category breakdown
- API success rate
- Uptime duration
- Scaling statistics
- Recent scaling events

**Use**: Generate daily summary email

### Alert Triggers File Reading

**Contains**:
- Triggered alert flags
- Alert-specific data
- Timestamps

**Use**: Process pending alerts

### Scaling State File Reading

**Contains**:
- Current tier
- Scaling history
- Tier time distribution
- Error counts

**Use**: Check for scaling anomalies

### Shutdown Flag Check

**Purpose**: Detect clean shutdown vs crash

**Method**: Check for shutdown flag file existence

**Logic**:
- Flag present: Clean shutdown, no alert
- Flag absent: Likely crash, send alert

## System 4: Bot Health Monitor

**Problem**: Detect when bot crashes

**Solution**: Monitor heartbeat freshness

### Health Check Strategy

**Heartbeat staleness**:
- Read heartbeat file
- Check timestamp age
- If age > timeout: Bot crashed
- If age <= timeout: Bot healthy

**Timeout threshold**: Reasonable duration (accounts for processing delays)

### Health Status Types

**HEALTHY**:
- Heartbeat recent
- Include current metrics

**CRASHED**:
- Heartbeat stale
- Include last known state

**UNKNOWN**:
- Heartbeat file missing
- Bot may not have started

### Crash Detection Data

**When crashed, extract**:
- Last heartbeat timestamp
- Events processed before crash
- Queue size at crash
- Uptime before crash
- Current tier at crash
- Consecutive API failures

**Purpose**: Debug why crash occurred

### Clean Shutdown Handling

**Check shutdown flag first**:
- If flag present: Not a crash
- Remove flag, no alert
- Return early

**Prevents**: False crash alerts on intentional shutdown

## System 5: Scaling Anomaly Detector

**Problem**: Detect unhealthy scaling patterns

**Solution**: Pattern analysis on scaling state

### Anomaly Types Detected

#### Stuck at High Tier
- **Condition**: High tier for extended duration
- **Threshold**: Hours at high tier
- **Indicates**: Sustained high load or leak

#### Excessive Oscillation
- **Condition**: Too many scaling events in short time
- **Threshold**: Many events in short window
- **Indicates**: Unstable load or bad thresholds

#### Rapid Tier Jumps
- **Condition**: Multiple tier levels in one jump
- **Threshold**: Jump size and frequency
- **Indicates**: Sudden load spike

#### High Tier With Failures
- **Condition**: High tier with many API errors
- **Indicates**: API struggling, should scale down

### Anomaly Data Structure

**Each anomaly contains**:
- Type identifier
- Severity level
- Metrics (tier, duration, count)
- Descriptive message
- Recommendation for fix

### Analysis Method

**Process**:
1. Read scaling state and heartbeat
2. Calculate current tier duration
3. Analyze recent scaling history
4. Check for each anomaly pattern
5. Return list of detected anomalies

**Returns**: List of anomaly dictionaries

## System 7: Alert Manager (Main Orchestrator)

**Problem**: Coordinate all monitoring activities

**Solution**: Main loop that runs all checks

### Initialization

**Configuration from environment**:
- State file directory
- Gmail credentials paths
- Sender and recipient emails
- Cooldown durations for each alert type

**Component creation**:
- Gmail sender (with authentication)
- Rate limiter (with cooldowns)
- State file reader
- Health monitor
- Anomaly detector
- Summary collector

### Watchdog Main Loop

**Runs continuously**:
1. Check bot health (crash detection)
2. Check alert triggers (pending alerts)
3. Check scaling anomalies
4. Check if time for daily summary
5. Sleep for interval
6. Repeat

**Interval**: Regular monitoring cycle

### Bot Health Checking

**Process**:
1. Call health monitor
2. If status CRASHED:
   - Check for clean shutdown flag
   - If flag: Remove and skip alert
   - If no flag: Send crash alert (if not on cooldown)
3. If status HEALTHY: No action
4. If status UNKNOWN: No action (bot not started yet)

### Alert Trigger Processing

**Process**:
1. Read alert triggers file
2. For each triggered alert:
   - Check if on cooldown
   - If not: Send alert email
   - Clear trigger flag
   - Save updated triggers file

**Alert types**:
- Queue overflow: Events lost
- API failure: Consecutive failures

### Scaling Anomaly Checking

**Process**:
1. Call anomaly detector
2. If anomalies detected:
   - Check if on cooldown
   - If not: Send anomaly alert
   - Include all detected anomalies

### Daily Summary Scheduling

**Time-based trigger**:
- Check current hour (in target timezone)
- Compare to configured summary hour
- Check if already sent today
- If time and not sent: Send summary

**Date tracking**: Prevent duplicate summaries same day

### Email Alert Templates

#### Crash Alert
- **Sections**:
  - Critical: Bot crashed notice
  - Last known state
  - Recommended actions
- **Priority**: Critical
- **Subject**: Emergency crash detected

#### Queue Overflow Alert
- **Sections**:
  - Warning: Events lost
  - Queue statistics
  - Possible causes
  - Sample lost events
- **Priority**: High
- **Subject**: Critical queue overflow

#### API Failure Alert
- **Sections**:
  - Error: API failure details
  - Consecutive failure count
  - Recommended actions
- **Priority**: High
- **Subject**: API failure detected

#### Scaling Anomaly Alert
- **Sections**:
  - Warning: Anomaly detected
  - Current system state
  - Anomaly details
  - Recommendations
  - Recent scaling events
- **Priority**: High
- **Subject**: Scaling anomaly

#### Daily Summary Email
- **Sections**:
  - Health report header
  - Main statistics
  - Scaling summary
  - Top event categories
  - Recent scaling events
- **Priority**: Normal
- **Subject**: Daily health report

### Trigger File Updates

**Process**:
1. Read current triggers
2. Clear processed trigger flags
3. Write updated triggers back

**Atomicity**: Use state file writer utilities

## Key Design Decisions

### Why Independent Process
- Survives bot crashes
- Can detect when bot dies
- Uses different communication channel (email vs platform)

### Why Gmail API
- Reliable delivery
- No SMTP configuration
- OAuth2 security
- Supports HTML formatting

### Why State Files
- Simple IPC mechanism
- Human-readable for debugging
- Survives crashes
- No complex messaging needed

### Why Rate Limiting
- Prevent alert fatigue
- Different urgency = different cooldowns
- Persistent state across restarts

### Why HTML Emails
- Better readability
- Color coding for urgency
- Professional appearance
- Tables for statistics

### Why Daily Summary
- Regular health check
- Proactive monitoring
- Historical tracking
- Catch slow degradation
