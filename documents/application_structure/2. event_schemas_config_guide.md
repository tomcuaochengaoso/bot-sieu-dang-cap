# Event Schema Configuration File - Technical Concepts Guide

**Build Order: #2 (Configuration Foundation)**  
**Dependencies**: None (pure configuration file)  
**Format**: YAML structured data  
**Purpose**: Define event naming patterns, validation rules, and business logic mappings

---

## File Purpose

This configuration file serves as the **single source of truth** for:
- Event naming patterns in target language
- Category classifications (flow categorization)
- Required vs optional properties for each event type
- Validation rules for data integrity
- Bot behavior configurations
- Special channel/role mappings

**Why YAML format**: Human-readable, supports comments, hierarchical structure, easy to modify without code changes

---

## Top-Level Structure

The file is organized into major sections:

1. Metadata and versioning
2. Common property templates
3. Event type definitions
4. Validation rules
5. Platform-specific mappings
6. Category definitions
7. Bot configurations
8. Change history documentation

---

## Section 1: Metadata

### Version Information
- **Schema version**: Tracking number for configuration evolution
- **Schema format identifier**: Declares what system uses this schema
- **Purpose**: Allows detecting incompatible schema versions

### Version Tracking Benefits
- Prevents using outdated configuration with new code
- Documents breaking changes
- Enables migration strategies
- Helps debug issues ("which schema version was running?")

## Section 2: Common Property Templates

### Purpose
Avoid repetition by defining reusable property sets that multiple event types share

### Template Categories

#### Base User Properties Template
- **Contains**: User identification properties
- **Used by**: All events involving user actions
- **Examples**: User identifier, display name, server context
- **Pattern**: YAML anchor/alias for reuse

#### Base Channel Properties Template
- **Contains**: Channel context properties
- **Used by**: Events occurring in channels
- **Examples**: Channel identifier, channel name, channel type

#### Base Message Properties Template
- **Contains**: Message-specific properties
- **Used by**: Message-related events
- **Examples**: Message identifier, content metadata, attachment info

#### Localized Event Properties Template
- **Contains**: Properties specific to localization system
- **Used by**: All events
- **Examples**: Translated event name, flow category, event type classification

#### Correlation Properties Template
- **Contains**: Properties for linking user actions to bot responses
- **Used by**: Bot interaction events
- **Examples**: Correlation type, confidence score, triggering user

#### Role Change Properties Template
- **Contains**: Properties for role modifications
- **Used by**: Role addition/removal events
- **Examples**: Role name, action type, target user

### Template Reuse Technique
- **YAML anchors**: Define once with `&anchor_name`
- **YAML aliases**: Reference with `*anchor_name`
- **Composition**: Combine multiple templates into single event definition
- **Benefit**: Change in one place propagates everywhere

## Section 3: Event Type Definitions

**Structure**: Each event type has its own subsection

### Per-Event Configuration

#### Naming Pattern
- **Static name**: Fixed event name for simple events
- **Dynamic pattern**: Template with variables for context-dependent names
- **Variable substitution**: Placeholders replaced at runtime
- **Examples**: Channel names, bot names, forum names embedded in event name

#### Category Classification
- **Static category**: Fixed flow category
- **Dynamic category pattern**: Template that generates category from context
- **Purpose**: Groups related events for analysis

#### Required Properties List
- **Mandatory fields**: Must be present for event to be valid
- **Validation enforcement**: Event rejected if missing required property
- **Uses templates**: References common property templates by alias

#### Optional Properties List
- **Nice-to-have fields**: Enrichment data when available
- **No validation failure**: Event still valid without these
- **Context-dependent**: May only apply in certain scenarios

#### Description Field
- **Human documentation**: Explains when this event fires
- **Examples included**: Shows typical scenarios
- **Helps maintainers**: Understand business logic without reading code

### Event Type Categories Covered

#### Onboarding Flow Events
- **Purpose**: Track user onboarding journey
- **Examples**: Special role assignments, welcome channel interactions, tutorial completion
- **Category pattern**: Onboarding-specific flow

#### General Communication Events
- **Purpose**: Track usage of main communication channels
- **Examples**: General chat usage, emoji reactions, sticker sends
- **Category pattern**: Communication feature flow

#### Announcement Events
- **Purpose**: Track announcement channel engagement
- **Examples**: Following announcement channels, webhook subscriptions
- **Category pattern**: Announcement viewing flow

#### Gaming/Activity Events
- **Purpose**: Track gaming bot interactions
- **Examples**: Game-specific bot usage, gaming thread messages
- **Category pattern**: Gaming activity flow
- **Special handling**: Some bots tracked by thread, others by correlation

#### Voice Channel Events (Two Subcategories)
- **Fixed voice channels**: Permanent server channels
  - **Category pattern**: Fixed voice flow
  - **Detection**: Based on category membership
- **Custom voice channels**: User-created temporary channels
  - **Category pattern**: Custom voice flow
  - **Detection**: Based on category non-membership

#### Text Channel Events
- **Purpose**: Track text channel interactions
- **Examples**: Messages, edits, deletions, reactions
- **Category pattern**: Text channel flow

#### Forum Events
- **Purpose**: Track forum usage
- **Examples**: Thread creation, replies, reactions, follows
- **Category pattern**: Dynamic based on forum name
- **Special**: Category name includes specific forum

#### Role Management Events
- **Purpose**: Track role changes
- **Examples**: Role additions, role removals
- **Category pattern**: Role management flow
- **Special**: Generates multiple events (role event + potential special trigger)

#### Bot Interaction Events
- **Purpose**: Track bot usage with correlation
- **Subcategories**: 
  - Correlation-tracked bots (link user to bot response)
  - Thread-tracked bots (identified by thread location)
  - Settings bots (configuration interactions)
- **Naming patterns**: May include location suffixes or bot names

#### Special Location Events
- **Purpose**: Track special triggers
- **Examples**: Room creation triggers, special thread activities
- **Category pattern**: Special action flows

### Dynamic Naming Examples

**Pattern with channel variable**:
- Template: "Action in channel {channel_name}"
- Runtime: channel_name = "voice-lobby"
- Result: "Action in channel voice-lobby"

**Pattern with forum variable**:
- Template: "Post in {forum_name}"
- Category template: "Interact with {forum_name}"
- Runtime: forum_name = "help-forum"
- Result: Event "Post in help-forum" in category "Interact with help-forum"

**Pattern with bot location suffix**:
- Base: "Use game bot"
- Suffix determination: Check if in specific thread
- Result: Either "Use game bot in thread" or "Use game bot outside thread"

## Section 4: Validation Rules

### Purpose
Define data type requirements and constraints for event properties

### Rule Types

#### Type Validation
- **String fields**: Must be text
- **Integer fields**: Must be whole numbers
- **Boolean fields**: Must be true/false
- **Float fields**: Must be decimal numbers
- **Array fields**: Must be lists
- **Object fields**: Must be structured data

#### Format Validation
- **Timestamp format**: Must follow timestamp standard (ISO format)
- **Identifier format**: Must be valid identifier string
- **Enumeration**: Must be one of allowed values

#### Constraint Validation
- **Minimum value**: Numeric floor
- **Maximum value**: Numeric ceiling
- **Minimum length**: String/array size floor
- **Maximum length**: String/array size ceiling
- **Pattern matching**: Must match regular expression

#### Conditional Validation
- **Required when**: Field required only if another field has specific value
- **Mutual exclusivity**: Only one of several fields can be present
- **Dependency**: Field X requires field Y to also be present

### Key Field Validations

#### User Identifier Fields
- **Type**: String (important: prevents precision loss with large numbers)
- **Why string**: Platform identifiers are very large integers
- **Precision issue**: Some JSON parsers lose precision on large integers
- **Solution**: Always store as string

#### Timestamp Fields
- **Type**: String
- **Format**: Standard timestamp format
- **Required**: Yes for all events
- **Purpose**: Event chronology

#### Localized Event Name
- **Type**: String
- **Required**: Yes
- **Minimum length**: Must not be empty
- **Purpose**: Primary event identifier in analytics

#### Flow Category
- **Type**: String
- **Required**: Yes
- **Allowed values**: Must be from predefined list
- **Purpose**: Event grouping and analysis

#### Event Type
- **Type**: String
- **Required**: Yes
- **Purpose**: Technical event classification

#### Trackability Flags
- **Type**: Boolean
- **Default value**: True
- **Purpose**: Mark events as trackable or not

#### Correlation Fields
- **Type**: String for identifiers, Float for confidence
- **Required when**: Only when correlation applies
- **Conditional**: If correlation type present, other correlation fields required

#### Role Fields
- **Required when**: Only for role-related events
- **Pattern**: Role name must exist, action must be from allowed set

### Validation Error Handling
- **Returns**: List of validation errors (empty if valid)
- **Early termination**: Stop processing invalid events
- **Logging**: Record what validation failed
- **Purpose**: Data quality assurance

### Per-Category Information

#### Category Name
- **Identifier**: Unique category name in target language
- **Purpose**: Groups related user journeys

#### Description
- **Purpose**: Human explanation of what this category tracks
- **Examples**: What events fall into this category

#### Event List
- **Contains**: Which event types belong to this category
- **Purpose**: Quick reference for category contents

#### Special Flags
- **Pattern events**: Whether events use dynamic naming
- **Role-specific**: Whether category is role-management related
- **Tracking method**: How events in this category are detected
  - Fixed channel list
  - Category-based (dynamic discovery)
  - Thread-based
  - Hybrid

### Category Types

#### Onboarding Category
- **Tracks**: User onboarding process
- **Events**: Tutorial interactions, special role triggers, welcome interactions

#### Communication Category
- **Tracks**: General communication feature usage
- **Events**: Chat, emoji, stickers, GIFs in main channels

#### Announcement Category
- **Tracks**: Announcement engagement
- **Events**: Following announcement channels

#### Gaming Category
- **Tracks**: Gaming and bot activities
- **Events**: Bot interactions, game threads, uncorrelated messages

#### Room Creation Category
- **Tracks**: Voice room creation
- **Events**: Creation trigger actions

#### Settings Category
- **Tracks**: Configuration bot usage
- **Events**: Settings bot interactions

#### Fixed Voice Category
- **Tracks**: Permanent voice channel usage
- **Events**: Join, leave, message, react in fixed channels
- **Detection**: Category membership check

#### Custom Voice Category
- **Tracks**: User-created voice channel usage
- **Events**: Join, leave, message, react in custom channels
- **Detection**: Category non-membership check

#### Text Channels Category
- **Tracks**: General text channel usage

#### Role Updates Category

- **Pattern**: Each forum has its own category
- **Naming**: Category name includes forum identifier
- **Events**: Thread creation, replies, reactions specific to that forum
- **Detection**: Category-based with automatic discovery

## Section 7: Bot Configuration

### Purpose
Define which bots are tracked and how

### Bot Categories

#### Correlation-Tracked Bots
- **Detection method**: Links user message to bot response within time window
- **Time window**: Short duration for correlation validity
- **Confidence calculation**: Based on timing between user action and bot response
- **Naming pattern**: May include location context (in/out of specific thread)
- **Examples**: Interactive game bots, command bots

##### Per-Bot Configuration
- **Bot identifier**: Platform's unique bot identifier (as string)
- **Bot name**: Human-readable name
- **Localized name**: Bot name in target language
- **Event name**: What to call interactions with this bot
- **Flow category**: Which category these events belong to
- **Thread identifier**: Associated thread (if any)
- **Use correlation**: Boolean flag

##### Special: Pending Message Timeout
- **Purpose**: Handle cases where user messages bot but bot doesn't respond
- **Mechanism**: Track user messages, start timeout timer
- **Timeout duration**: Short wait period
- **On timeout**: Generate uncorrelated event
- **On bot response**: Cancel timeout, generate correlated event
- **Event names**: Different for correlated vs uncorrelated

#### Thread-Tracked Bots
- **Detection method**: User messages in specific thread
- **No correlation**: Don't track who triggered bot
- **Identification**: Thread membership alone
- **Simpler**: No timeout mechanism needed
- **Examples**: Persistent game threads, information bots

##### Per-Bot Configuration
- **Bot identifier**: Platform bot identifier
- **Bot name**: Human and localized names
- **Thread identifier**: Which thread indicates this bot
- **Event name**: What to call usage
- **Flow category**: Event grouping

#### Settings Bots
- **Purpose**: Configuration and room management
- **Use correlation**: Track who used settings
- **Category**: Special settings category

### Correlation Logic Details

#### Correlation Window
- **Duration**: Short time window for matching
- **Purpose**: User action must be close to bot response

#### Confidence Calculation
- **Formula**: Based on time difference between user and bot
- **Range**: From no confidence to full confidence
- **Minimum threshold**: Below this, don't correlate
- **Purpose**: Avoid false correlations

#### Location Tracking (Simplified)
- **Old approach**: Complex special thread tracking (removed)
- **New approach**: Simple thread identifier comparison
- **Suffix logic**: Check if bot response thread matches specific thread
- **Result**: Binary suffix (in designated thread or not)

#### User Message Buffer
- **Structure**: Store recent user messages per channel
- **Capacity**: Limited number of messages
- **Cleanup**: Remove old messages
- **Purpose**: Quick lookup when bot responds

## Section 8: Special Channel/Role Configurations

### Purpose
Define channels and roles with special tracking behavior

### Marketing Role Set
- **List**: Specific role names
- **Purpose**: Trigger special onboarding event when assigned
- **Dual events**: Generate both role change event AND special onboarding event
- **Category**: Onboarding category for special event

### Mod Message Channels
- **Structure**: Map of channel identifiers to configuration
- **Per-channel**: 
  - Channel name and localized name
  - Special event name for mod updates
  - List of authorized sender identifiers
- **Purpose**: Track when moderators update important channels
- **Category**: Onboarding category

### Special Thread Map
- **Structure**: Map of thread identifiers to configuration
- **Per-thread**:
  - Thread name and localized name
  - Special event name
  - Flow category
- **Purpose**: Threads with unique tracking behavior
- **Example**: Specific game threads not tracked by bot correlation

### General Chat Channel
- **Single identifier**: The main chat channel
- **Purpose**: Separate tracking for general chat usage
- **Special events**: Different event names for chat vs other channels

### Room Creation Trigger Channel
- **Single identifier**: Voice channel that triggers room creation
- **Purpose**: Detect when users trigger room creation feature
- **Event**: Special room creation event instead of normal voice join

### Category-Based Tracking

#### Forum Category Tracking
- **Method**: Automatic discovery based on category membership
- **Tracked categories**: List of category identifiers
- **Benefits**: New forums automatically tracked when added to category
- **Cache**: Store category names for performance

#### Voice Category Tracking
- **Method**: Distinguish fixed vs custom by category
- **Tracked category**: Specific category for fixed channels
- **Detection**: In category = fixed, not in category = custom
- **Benefits**: No need to list every channel individually

## Configuration Management Patterns

### Separation of Concerns
- **Configuration**: In YAML file (this file)
- **Logic**: In code files
- **Benefit**: Change behavior without code changes

### Single Source of Truth
- **Event names**: Only defined here
- **Categories**: Only defined here
- **Validation**: Only defined here
- **Benefit**: No duplication, no conflicts

### Declarative vs Imperative
- **Declarative**: Describe WHAT, not HOW
- **Example**: "Event requires these properties" not "Check if properties exist"
- **Benefit**: Easier to understand and modify

### Validation at Configuration Level
- **Schema validation**: Ensure YAML is well-formed
- **Completeness checks**: All required sections present
- **Consistency checks**: No conflicting rules
- **Benefit**: Catch errors before runtime

## Testing Strategies for Configuration

### Schema Validation Testing
- Load YAML file
- Verify all required sections present
- Check no syntax errors
- Validate structure matches expected format

### Completeness Testing
- Every event type has all required fields defined
- Every category referenced by events actually exists
- Every template referenced by events is defined
- All bot identifiers are unique

### Consistency Testing
- Required properties don't conflict with optional properties
- Flow category names match between sections
- Event types match between definitions and validation rules

### Regression Testing
- Test events from previous version still work
- Deprecated events handled gracefully
- New events validate correctly

## Key Design Principles

**Declarative Configuration**:
- Describe desired outcome, not implementation steps

**DRY (Don't Repeat Yourself)**:
- Templates eliminate duplication
- YAML anchors/aliases for reuse

**Open/Closed Principle**:
- Open for extension (add new events easily)
- Closed for modification (existing events stable)

**Schema Evolution**:
- Version tracking
- Backward compatibility considerations
- Migration path documentation

**Human-Readable**:
- YAML format chosen for readability
- Comments allowed
- Hierarchical structure matches mental model
- Names are descriptive, not cryptic
