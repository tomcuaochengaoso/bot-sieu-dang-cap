# Platform Event Listener Module - Technical Concepts Guide

**Build Order: #5 (Data Collection Layer)**  
**Dependencies**: Event processor (forwards events to), Event queue (indirectly via processor)  
**Purpose**: Listen to platform events, extract data, forward for processing

---

## Module Purpose

This module is the **data collection layer**:
- Connects to platform using platform's official library
- Registers event handlers for all relevant events
- Extracts raw event data from platform objects
- Forwards data to event processor
- Provides minimal processing (pure data extraction)

**Core Responsibility**: "Something happened on platform" → "Here's the raw data, processor handle it"

---

## Architecture: Pure Event Forwarding

**Design Philosophy**: Thin wrapper around platform library

### Separation of Concerns
- **This module**: Listen and extract
- **Event processor**: Identify and transform
- **Queue handler**: Buffer and transmit

**Benefit**: Changes to business logic don't affect listening code

---

## Component 2: Event Handler Setup

**Pattern**: Define all event handlers in single method

**Structure**: For each platform event type, define async callback

### Event Categories Handled

- Reaction removed from message

#### Voice Channel Events
- User joined voice channel
- User left voice channel  
- User switched voice channels

#### Member/Role Events
- Member role changes (individual role updates)

#### Thread Events
- Thread created
- Thread deleted
- Thread member joined (follow tracking)

#### Webhook/Announcement Events
- Webhook updates (announcement follows)

#### Scheduled Event Events
- User interested in event
- User not interested in event

#### Message Events
- Message sent (user or bot)
- Message edited

#### Invite Events
- Invite created

- Member joined server
- Member left server

#### Channel Management Events
- Channel created

### Handler Registration Pattern

**Platform decorator**: `@client.event` (Discord.py's official decorator)
**Async function**: All handlers are async
**Callback structure**: Platform calls when event occurs

## Component 3: Event Handlers - Details

### Reaction Event Handlers

**On reaction add**:
1. Check if user is bot (skip if yes)
2. Extract emoji data
3. Extract message context
4. Check if in thread (add thread_id)
5. Forward with event type "reaction_added"

**On reaction remove**:
- Same process as add

**Emoji extraction**: Call helper method (handles custom vs unicode)

**Thread detection**: Check if message channel has parent

### Voice Channel Event Handlers

**On voice join**:
1. Check if user is bot (skip)
2. Extract user data
3. Extract channel data
4. Include voice session ID
5. Forward 

**On voice leave**:
- Same process as join

**On voice switch**:
1. Detect: before channel ≠ after channel
2. Extract both channels (from and to)
3. Include previous channel data
4. Forward 

**Enhanced forwarding**: Includes previous channel context

### Role Change Event Handlers

**On member update**:
1. Detect: role list changed
2. Calculate added roles (in after, not in before)
3. Calculate removed roles (in before, not in after)
4. **For each added role**: Forward separate event
5. **For each removed role**: Forward separate event

**Critical**: Individual events per role
**Purpose**: Event processor can handle each independently

### Thread Event Handlers

**On thread create**:
1. Extract thread data
2. Extract parent channel type
3. Forward with parent context
4. Processor filters (only forums tracked)

**On thread delete**:
- Same process

**On thread member join**:
1. Skip if bot itself joined
2. Get user who joined
3. Extract thread and parent info
4. Get parent channel type
5. Forward 

**Parent channel type**: Critical for processor filtering

### Webhook/Announcement Event Handlers

**On webhooks update**:
1. Check if announcement channel
2. Wait briefly (audit log delay)
3. Query audit log for webhook creation
4. Filter: recent entries only
5. Verify: webhook for this channel
6. Check: follower webhook type
7. Extract user who created webhook
8. Forward 

**Complexity**: Requires audit log permission
**Timing**: Must wait for audit log to update
**Filtering**: Many checks to confirm it's announcement follow

### Scheduled Event Handlers

**On user add to event**:
1. Extract event data
2. Extract user data
3. Forward 

**On user remove from event**:
- Same process

### Message Event Handlers

**On message**:
1. Check if bot (handle separately)
2. Extract message data
3. **Extract thread context** (thread_id AND thread_name)
4. Detect content type (attachment, sticker, gif)
5. Forward with appropriate event type

**Thread extraction**: Critical for bot monitoring

**Bot messages**: Special path for correlation

**Content detection**:
- Has attachments: Note in data
- Has stickers: Create sticker event
- Contains GIF indicators: Create GIF event
- Otherwise: Regular message event

**On message edit**:
1. Check not bot
2. Extract edit data
3. Include thread context
4. Forward 

**On message delete**:
- Same process

### Invite Event Handlers

**On invite create**:
1. Extract inviter
2. Extract channel
3. Include invite metadata (code, URL, limits)
4. Forward 

### Server Event Handlers

**On member join**:
1. Extract member data
2. Include join timestamp
3. Forward 

**On member leave**:
- Same process

### Channel Management Handlers

**On channel create**:
1. Extract channel data
2. Include channel type
3. Include parent category if exists
4. Forward

**On channel delete**:
- Same process

## Component 4: Data Extraction Methods

**Purpose**: Convert platform objects to standard dictionaries

### Emoji Data Extraction

**Handles two types**:
1. Custom emoji (platform-specific object)
2. Unicode emoji (string)

**Custom emoji extraction**:
- Emoji identifier (as string)
- Emoji name
- Animated flag
- Primary representation (name)

**Unicode emoji extraction**:
- Unicode character
- Mapped name (from lookup table)
- Primary representation (character)

**Partial emoji**: Fallback handling

**Returns**: Standardized dictionary

### Unicode to Name Mapping

**Purpose**: Convert common unicode emojis to names

**Method**: Lookup table
**Coverage**: Common emojis
**Fallback**: Return unicode if not in table

### Timestamp Generation

**Method**: Get current time
**Format**: Platform's standard timestamp
**Use**: Event timing

## Component 5: Event Forwarding

**Pattern**: All events flow through single forwarding method

### Main Forwarding Method

**Takes**:
- Event type (string)
- Raw data (keyword arguments)

**Process**:
1. Call event processor
2. Processor returns list of processed events
3. For each processed event:
   - Add to event queue
   - Increment counter
   - Log event

**Error handling**: Catch and log exceptions

### Bot Message Forwarding

**Separate path**: Bot messages handled differently

**Takes**:
- Bot message object
- Thread context (if applicable)

**Process**:
1. Extract thread_id and thread_name
2. Determine parent channel (thread parent or channel)
3. Forward as regular message with bot flag
4. Include bot-specific data

**Purpose**: Enable bot correlation in processor

## Data Flow Example

### User sends message in thread
on_message, on_reaction_add 
- These are Discord.py's official API event names
message.author, message.channel, reaction.emoji 
- Discord.py's standard object properties
1. **Platform library**: Detects message event
2. **Handler**: `on_message` callback triggered
3. **Extract**: 
   - User data from message.author
   - Channel data from message.channel
   - Thread data from message.channel (if thread)
   - Thread name for context
4. **Forward**: Send to `_forward_raw_event`
5. **Process**: Event processor identifies event
6. **Queue**: Processed events added to queue
7. **Log**: Print event processed

### User reacts to message in forum thread

1. **Platform library**: Reaction add event
2. **Handler**: `on_reaction_add` callback
3. **Extract**:
   - User data from user parameter
   - Emoji data from reaction.emoji
   - Message data from reaction.message
   - Thread context from message.channel
4. **Forward**: Send with thread_id included
5. **Process**: Processor identifies as forum reaction
6. **Queue**: Event queued for batching

## Thread Context Handling

**Critical for bot monitoring**: Thread_id AND thread_name

### Why Thread Name Matters

**Problem**: Uncorrelated events need meaningful names

**Solution**: 
- Extract thread name from channel object
- Include in forwarded data
- Processor uses for event naming
- Result: "Message in {thread_name}" instead of "Message in thread"

### Thread Detection

**Check**: Does channel have parent attribute?
**If yes**: Channel is thread
**Extract**: thread_id from channel.id, thread_name from channel.name (Discord.py's standard object properties)

**Include**: Both in forwarded data

## Bot vs User Message Handling

### User Messages
- Forward to processor
- Processor adds to correlation buffer
- May trigger pending timeout

### Bot Messages
- Forward with bot flag
- Processor attempts correlation
- Different event naming

**Distinction**: Made at platform library level (bot attribute)

## Error Handling Strategy

**All handlers**: Wrapped in try-catch (implicit via forwarding)

**Forwarding errors**: Caught and logged

**Continue on error**: Single bad event doesn't stop listening

**Log details**: Print error for debugging

## Platform-Specific Considerations

### Permission Requirements

**Audit log access**: Needed for announcement follow tracking
**Intent permissions**: Need all intents enabled
**Server permissions**: Bot needs appropriate server permissions

**Graceful degradation**: Print warning if permission missing

### Rate Limiting

**Platform enforces**: Rate limits on API calls
**Not handled here**: Platform library handles rate limiting
**Application layer**: No additional rate limit logic needed

### Reconnection

**Platform library**: Handles reconnection automatically
**Application**: Just call start, library manages connection

## Key Design Decisions

### Why Minimal Processing Here
- Keep this layer thin
- Business logic in processor (easier to change)
- This layer just extracts data

### Why Include Thread Name
- Critical for uncorrelated event naming
- Can't reconstruct later
- Must capture at extraction time

### Why Individual Role Events
- Simpler processor logic
- Each role change independent
- Can have different business meanings

### Why Check Parent Channel Type
- Processor needs to filter threads
- Can't determine later without platform access
- Extract context at source

### Why Bot Separate Path
- Different correlation logic
- Different timeout handling
- Cleaner separation
