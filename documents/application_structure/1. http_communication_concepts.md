# HTTP API Communication Module - Technical Concepts Guide

**Purpose**: Send data to external analytics platform via HTTP  
**Core Challenge**: Reliable data transmission despite network failures, rate limits, and size constraints

---

## Architecture Overview: Multiple Technical Components

This module solves API communication through several distinct technical responsibilities, built in this order:

1. Error categorization system
2. Communication interface abstraction
3. HTTP execution engine
4. Automatic retry orchestration
5. Data transformation and validation
6. High-level API orchestrator
7. Pre-configured instance creator
8. Metrics collection system

---

## Component 1: Error Categorization

**Problem**: Not all failures should be handled the same way

**Solution**: Error hierarchy with multiple categories

### Base Error Type
- **Purpose**: Common ancestor for all API-related failures
- **Technique**: Exception inheritance hierarchy
- **Usage**: Allows catching all API errors with single handler

### Temporary Failure Type
- **Indicates**: Problem might resolve itself (server overload, network hiccup)
- **Should**: Trigger automatic retry
- **Examples**: 5xx server errors, network timeouts, rate limit responses

### Permanent Failure Type  
- **Indicates**: Problem won't resolve by retrying (bad request, authentication)
- **Should**: Fail immediately without retry
- **Examples**: 4xx client errors, invalid data format

### Size Violation Type
- **Indicates**: Data payload exceeds limits
- **Should**: Trigger batch splitting or rejection
- **Examples**: Request larger than platform size limit, too many items in batch

**Key Technique**: Exception types encode retry decision logic

## Component 2: Communication Interface

**Problem**: Want to swap HTTP implementations (production vs testing) without changing code

**Solution**: Define abstract contract for HTTP operations

### Interface Definition
- **Technique**: Protocol/Interface pattern from object-oriented design
- **Specifies**: What operations must exist, not how they work
- **Benefits**: Can create fake implementations for testing

### Required Operation
- **Signature**: Asynchronous POST request
- **Inputs**: Destination URL, request body as bytes, HTTP headers
- **Output**: Parsed response as structured data
- **Contract**: Must raise appropriate error types on failure

**Key Technique**: Dependency Inversion Principle - depend on abstractions, not implementations

## Component 3: Production HTTP Engine

**Problem**: Actually execute HTTP requests with proper error handling

**Solution**: Concrete implementation using async HTTP library

### Resource Management
- **Technique**: Lazy initialization pattern
- **Creates**: HTTP session on first use
- **Reuses**: Same session for subsequent requests
- **Cleanup**: Explicit session closure method

### Request Execution Logic
- **Step 1**: Ensure session exists (create if needed)
- **Step 2**: Track request in counter (for statistics)
- **Step 3**: Execute POST with timeout
- **Step 4**: Examine response status code
- **Step 5**: Categorize error appropriately
- **Step 6**: Parse successful response

### Status Code Handling Strategy
- **Server error codes**: Raise temporary failure (server problem, retry later)
- **Rate limit response**: Raise temporary failure with rate limit signal
- **Payload too large response**: Raise permanent failure (won't work even if retry)
- **Other client errors**: Raise permanent failure (client mistake)
- **Success responses**: Attempt JSON parsing
- **Network errors**: Raise temporary failure

### Response Parsing
- **Primary**: Parse as JSON
- **Fallback**: If JSON parsing fails but status indicates success, treat as success anyway
- **Error**: If parsing fails on error status, raise permanent failure

### Statistics Tracking
- **Counters**: Total requests, total errors
- **Calculation**: Success count derived from difference

**Key Techniques**: 
- Connection pooling through session reuse
- Status code semantic interpretation
- Graceful degradation on parsing failures

## Component 4: Retry Orchestration

**Problem**: Transient failures should retry automatically, but not forever

**Solution**: Exponential backoff with jitter

### Configuration Parameters
- Maximum retry attempts before giving up
- Initial delay between retries
- Maximum delay cap
- Whether to add randomness to delays

### Delay Calculation Algorithm
- **Formula**: Initial delay multiplied by exponential growth factor based on attempt number
- **Cap**: Never exceed maximum delay threshold
- **Jitter**: Add small random variance to prevent simultaneous retries
- **Example progression**: Short delay → moderate delay → longer delay → much longer delay → maximum delay (capped)

### Retry Execution Logic
- **Loop**: Attempt operation up to max attempts
- **Success**: Reset counters, return result immediately
- **Permanent failure**: Don't retry, raise immediately
- **Temporary failure**: Calculate delay, sleep, try again
- **All failed**: Raise last error encountered

### State Tracking
- **Counter**: Track how many retries occurred
- **Reset**: Clear counter on successful operation
- **Purpose**: Monitoring and debugging

**Key Techniques**:
- Exponential backoff (doubles each time)
- Jitter (randomness prevents thundering herd)
- Circuit breaking (stop on permanent failures)
- State reset (don't carry over retry count between operations)

## Component 5: Data Transformation and Validation

**Problem**: API has strict requirements for data format and size

**Solution**: Utility functions for formatting, estimation, validation, and splitting

### Platform Constraints (Constants)
- Maximum items per batch: Platform-defined reasonable limit
- Maximum total size: Platform-enforced megabyte limit
- Safe size threshold: Conservative limit below maximum (safety margin)
- Average item size estimate: Typical expected item size
- Minimum item size estimate: Smallest expected item size
- Maximum item size estimate: Largest expected item size

### Batch Encoding Operation
- **Input**: List of data items
- **Output**: Byte array (encoded JSON)
- **Process**:
  1. Check item count within limit
  2. Convert to JSON string (compact format, no whitespace)
  3. Encode as UTF-8 bytes
  4. Verify size within limit
  5. Return bytes or raise size violation

### Size Estimation Strategy
- **Small batches**: Quick multiplication estimate using average size
- **Medium batches**: Sample first few items, extrapolate with safety margin
- **Large batches**: Actually serialize and measure
- **Purpose**: Avoid creating oversized batches

### Safety Check Operation
- **Checks**: Both item count and estimated size
- **Threshold**: Uses conservative safe limit below absolute maximum
- **Returns**: Boolean indicating safety

### Batch Splitting Algorithm
- **Recursive approach**:
  1. Check if batch is safe
  2. If safe → return as single batch
  3. If too large → split in middle
  4. Recursively check each half
  5. Continue until all resulting batches are safe
- **Returns**: List of safe-sized batches

### Authentication Header Creation
- **Input**: API token
- **Process**:
  1. Append colon to token
  2. Encode as UTF-8
  3. Base64 encode
  4. Create Authorization header with "Basic" scheme
- **Additional headers**: Content-Type, Accept, User-Agent

### Validation Operation
- **Checks performed**:
  - Batch not empty
  - Item count within limit
  - Estimated size within limit
  - Each item is structured data (dictionary/object)
  - Each item has required "event" field
  - Each item has required "properties" field
  - Properties contain "time" field
  - Properties contain "distinct_id" field
- **Returns**: List of error messages (empty if valid)

**Key Techniques**:
- Lazy evaluation (estimate before full serialization)
- Sampling strategy (extrapolate from subset)
- Recursive divide-and-conquer (splitting)
- Base64 encoding for authentication
- Schema validation without formal schema definition

## Component 6: High-Level Orchestration

**Problem**: Coordinate HTTP, retry, and formatting into single cohesive operation

**Solution**: Orchestrator that uses other components

### Dependencies (Injected)
- HTTP communication engine (implementing interface)
- Retry policy engine
- API token
- API endpoint URL

### Configuration Constants
- Default timeout duration: Reasonable wait time for network operations
- Maximum retry attempts: Several attempts before giving up
- Base retry delay: Short initial wait period
- Maximum retry delay: Upper limit on wait time

### Statistics State
- Events sent successfully
- Batches sent successfully  
- Total bytes transmitted
- Failed batch count

### Main Operation: Send Batch
- **Inputs**: List of events, auto-split flag, validate flag
- **Process**:
  1. Early return if empty
  2. Run validation if requested
  3. Check if size is safe
  4. If oversized and auto-split enabled → call split sender
  5. Otherwise → call single batch sender
- **Returns**: Result dictionary with status and metrics

### Single Batch Sender (Internal)
- **Process**:
  1. Format batch to bytes
  2. Create authentication headers
  3. Append strict validation parameter to URL
  4. Define send operation (wraps HTTP call)
  5. Execute through retry policy
  6. Update statistics counters
  7. Process response
  8. Return result dictionary
- **Handles**: All error types appropriately

### Split Batch Sender (Internal)
- **Process**:
  1. Split oversized batch into chunks
  2. For each chunk:
     - Send via single batch sender
     - Collect result
     - Brief pause to avoid rate limit
  3. Aggregate all results
  4. Return combined summary

### Response Processor (Internal)
- **Checks**: Response for error indicators
- **Returns**: Structured result with:
  - Success/failure status
  - Events sent/failed count
  - Size metrics
  - API response
  - Timestamp

### Statistics Retrieval
- **Returns**: Dictionary containing:
  - Raw counters (events, batches, bytes, failures)
  - Calculated rates (success rate, failure rate)
  - Averages (per batch size in events and bytes)
  - Size in megabytes

**Key Techniques**:
- Dependency injection (receives engines via constructor)
- Composition over inheritance (uses other components)
- Strategy pattern (retry policy is swappable)
- Statistics aggregation (tracks metrics at orchestration level)
- Automatic batch splitting (handles oversized data transparently)

## Component 7: Pre-configured Instance Factory

**Problem**: Creating orchestrator requires multiple dependencies with correct configuration

**Solution**: Factory methods that hide construction complexity

### Production Configuration Creator
- **Inputs**: API token, endpoint URL
- **Creates**:
  1. HTTP engine with standard timeout
  2. Retry policy with aggressive retries (multiple attempts, exponential delays, with jitter)
  3. Orchestrator combining both
- **Returns**: Ready-to-use orchestrator
- **Use case**: Production environment

### Testing Configuration Creator
- **Inputs**: Mock HTTP engine, API token, endpoint URL
- **Creates**:
  1. Uses provided mock engine
  2. Retry policy with minimal retries (very few attempts, short delay, no jitter)
  3. Orchestrator combining both
- **Returns**: Test-ready orchestrator
- **Use case**: Unit testing

### High-Throughput Configuration Creator
- **Inputs**: API token, endpoint URL
- **Creates**:
  1. HTTP engine with extended timeout (longer than standard)
  2. Retry policy with fewer retries (reduced attempts, shorter delays)
  3. Orchestrator combining both
- **Returns**: Speed-optimized orchestrator
- **Use case**: High-volume scenarios where speed matters

**Key Techniques**:
- Factory pattern (encapsulate object creation)
- Configuration variants (different setups for different needs)
- Sensible defaults (pre-tuned for common scenarios)

## Component 8: Metrics Collection

**Problem**: Track transmission statistics across concurrent operations

**Solution**: Thread-safe statistics tracker

### State Storage
- Dictionary containing multiple counters (all start at zero)
- Asynchronous lock for thread safety

### Recording Operation
- **Inputs**: Events count, success flag, retry count, bytes count
- **Process** (with lock acquired):
  1. Increment total events counter
  2. Increment API calls counter
  3. Add to retry counter
  4. Add to bytes counter
  5. Increment success counter OR failure counter based on flag
- **Technique**: Lock ensures only one operation modifies counters at a time

### Statistics Retrieval
- **Process** (with lock acquired):
  1. Copy counter dictionary
  2. Release lock
  3. Calculate derived metrics:
     - Success rate (successful ÷ total)
     - Failure rate (failed ÷ total)
     - Average bytes per call (total bytes ÷ calls)
     - Megabytes sent (bytes converted to megabytes)
  4. Return enhanced dictionary

### Reset Operation
- **Process** (with lock acquired):
  1. Set all counter values to zero

**Key Techniques**:
- Reader-writer lock pattern (async lock)
- Copy-on-read (return copy not reference)
- Derived metrics (calculated from raw counters)
- Thread safety (lock prevents data races)

## Critical Implementation Considerations

**Async/Await Required**:
- All network operations must be asynchronous
- Blocking calls will freeze entire application
- Use async/await syntax throughout

**Base64 Authentication**:
- API expects Basic auth
- Format: token + colon, then base64 encode
- Include in Authorization header

**Compact JSON**:
- No whitespace in JSON encoding
- Minimizes transmission size
- Use separators parameter in JSON encoder

**Size Checking Essential**:
- API enforces hard size limit
- Exceeding causes rejection of entire batch
- Better to split than to fail

**Retry Jitter Important**:
- Multiple clients retrying simultaneously create load spikes
- Random delays spread retry load
- Prevents thundering herd problem

**Statistics at Every Layer**:
- HTTP engine tracks requests/errors
- Retry policy tracks retry counts
- Orchestrator tracks events/batches/bytes
- Each layer provides diagnostic value

**Resource Cleanup**:
- HTTP sessions must be explicitly closed
- Prevents resource leaks
- Important for long-running applications

**Error Type Precision**:
- Correctly categorizing errors is critical
- Permanent errors shouldn't waste time retrying
- Temporary errors need retry to succeed
- Misclassification causes wrong behavior
